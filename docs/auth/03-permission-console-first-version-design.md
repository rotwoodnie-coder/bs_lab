# 权限配置台第一版设计稿（方案评审版 / 可落地开发版）

> 存档位置：`docs/auth/03-permission-console-first-version-design.md`
>
> 适用范围：超级管理员权限配置、角色可见性控制、基础 `R/W` 操作控制
>
> 目标：先满足项目的基本要求，再逐步扩展到更细的权限治理能力。

---

## 0. 结论先行

本方案建议**分两期落地**：

- **Phase 1**：先把权限资源目录和角色授权数据模型建起来，让权限“可查、可汇总、可初始化”
- **Phase 2**：再把可视化授权页面、保存接口、菜单/路由/接口鉴权全部接通，让权限“可配置、可保存、可生效”

按照评审意见，本版方案做如下收敛：

- `sys_menu.menu_id` 改为**自增 INT 主键**
- `menu_code` 作为**唯一业务编码**
- 初始化数据**手工编写 SQL**，第一版**不做自动收集**
- `resolvePermissionCodes` 增加 `sys_role_menu_perm` 读取逻辑，输出 `PAGE_{menuCode}_READ / WRITE`
- `GET` 默认校验 `PAGE_xxx_READ`
- `POST/PATCH/PUT/DELETE` 默认校验 `PAGE_xxx_WRITE`
- 前端路由守卫使用 `withPermission` 或 hook，按 `PAGE_{menuCode}_READ` 控制进入

### 当前必须补齐的现实问题

现在这 7 个角色需要按**治理层级 + 业务层级**重新分层，否则会出现：

- 区管/校管/教研员页面范围混乱
- 教师页面被错误地套上组织管理权限
- 学生 / 家长和教师 / 教研员的业务边界不清
- 系统管理员页面数量与真实职责不匹配

### 推荐的角色分层逻辑

本项目建议把 7 个角色分成 **3 层治理 + 2 层业务**：

1. **系统治理层**
   - `Role_Sys_Admin`
   - 主要给开发、运维、系统配置使用
   - 菜单最多，但不代表“业务权限最高”，而是“系统能力最全”

2. **区域 / 学校治理层**
   - `Role_District_Admin`
   - `Role_School_Admin`
   - 主要负责组织结构、用户、角色、班级、基础资源治理
   - 校级只管理到校；区管覆盖全区

3. **业务执行层**
   - `Role_Researcher`
   - `Role_Teacher`
   - 主要负责教研、实验、题库、授课、班级教学执行
   - 教研员与教师属于同一业务链，只是权限范围更大

4. **学习使用层**
   - `Role_Student`
   - `Role_Parent`
   - 主要负责学习、任务、成长、家庭协作
   - 不应看到组织治理类菜单

### 这一分层的核心原则

- **系统管理员最强，但主要是系统配置与开发调试入口，不是业务审批中心**
- **区管 > 校管 是治理范围差异，不是页面数量越多越高**
- **教研员 > 教师 是业务权限边界更宽，不是组织管理权限更高**
- **学生 / 家长是纯使用层，不参与组织治理**
- **菜单数量不代表权限高低，最终以角色授权为准**

这样做的结果是：

- 不动现有身份链路
- 不改变 `data_role` 的字典职责
- 可以先满足项目基本要求
- 后续还能平滑扩展按钮级、字段级、审计级权限

## 1. 背景与问题定义

当前系统已经具备：

- 角色字典
- 用户-角色绑定
- 登录态中的角色识别
- 现有的硬编码权限码判断
- 角色管理页的基础展示能力

但还缺少一个真正可落地的“角色权限配置台”，核心缺口有三类：

1. **没有统一的资源目录**
   - 系统里有哪些页面、菜单、功能点，目前没有统一登记

2. **没有可保存的角色授权数据**
   - 目前权限更偏向代码常量和固定映射，不是数据库配置

3. **没有完整的前后端权限闭环**
   - 前端菜单、页面、按钮、后端接口没有统一按权限接入

因此，本方案的第一版目标不是做完整 IAM，而是先做一个**最小闭环的权限配置系统**。

---

## 2. 设计目标

第一版只回答四个问题：

1. 这个角色**能看到什么**
2. 这个角色**能进入什么页面**
3. 这个角色**能执行什么写操作**
4. 这个配置**能否保存并生效**

### 2.1 第一版必须达到的结果

- 超级管理员可按角色查看权限
- 超级管理员可按角色配置 `R/W`
- 配置结果可保存到数据库
- 菜单和路由可按权限显隐
- 修改类接口可按权限拦截
- 页面查询默认校验 `PAGE_xxx_READ`
- 页面写入默认校验 `PAGE_xxx_WRITE`
- 页面目录必须覆盖系统主要可操作入口

### 2.2 第一版不做的内容

- 不做复杂继承树
- 不做字段级权限
- 不做数据行级权限
- 不做按钮级的全量细分模型
- 不做权限缓存（Redis/JWT）
- 不做权限版本回滚
- 不做自动收集
- 不做复杂的审计回放台

## 3. 设计原则

### 3.1 保持现有身份链路不变

继续沿用现有身份模型：

- `data_role`：角色字典
- `sys_user_role`：用户与角色绑定
- `sys_user`：用户基础信息

**不在 `data_role` 中直接塞权限字段**，避免污染角色字典职责。

### 3.2 权限与业务解耦

权限台只负责：

- 资源目录
- 角色授权
- 权限保存
- 权限生效

不负责业务数据本身，不负责实验/教研/组织等业务逻辑。

### 3.3 第一版只做最小闭环

第一版只做：

- `R` = 读 / 看 / 进入
- `W` = 写 / 改 / 保存 / 删除等写操作

先保证能跑通，不追求一次到位。

### 3.4 资源目录第一版手工维护

评审意见要求第一版**不做自动收集**，因此采用：

- 手工编写 SQL 初始化 `sys_menu`
- 后续如果有需要，再补脚本或管理工具

## 4. 角色分类建议

## 4.1 三大管理员

### 系统管理员 `Role_Sys_Admin`

- 系统级最高权限
- 面向开发、运维、配置、兜底管理
- 拥有全量菜单与全量授权的能力
- 不建议作为日常业务角色使用

### 区管 `Role_District_Admin`

- 管全区范围
- 负责组织、学校、教师、角色、基础资源治理
- 可查看全区范围的统计和配置
- 可管理多个学校

### 校管 `Role_School_Admin`

- 只管理本校范围
- 负责学校内组织结构、教师、班级、基础业务配置
- 不越级管理整个区域

## 4.2 两大业务角色组

### 教研员 `Role_Researcher`

- 教研业务管理
- 实验、题库、教材、教研组
- 可覆盖多个学校或区域内的业务执行

### 教师 `Role_Teacher`

- 我的教学班级
- 我的课堂
- 我的任务
- 我的实验
- 我的题库使用

### 学生 `Role_Student`

- 学习与完成任务
- 实验库、闯关、成长足迹
- 不参与组织治理

### 家长 `Role_Parent`

- 任务协作
- 家庭实验室
- 孩子学习进度查看
- 不参与组织治理

### 6.4 初始化 SQL 建议

```sql
-- 示例：角色默认授权的初始化方式
-- 具体 menu_id 以 sys_menu 实际自增结果为准
INSERT INTO sys_role_menu_perm (seq_id, role_id, menu_id, can_read, can_write, status, create_time)
SELECT CONCAT('RM_', r.role_id, '_', m.menu_id) AS seq_id,
       r.role_id,
       m.menu_id,
       CASE WHEN r.role_id IN ('Role_Sys_Admin','Role_District_Admin','Role_School_Admin') THEN 1 ELSE 0 END AS can_read,
       CASE WHEN r.role_id IN ('Role_Sys_Admin','Role_District_Admin') THEN 1 ELSE 0 END AS can_write,
       'y', NOW()
FROM data_role r
JOIN sys_menu m ON 1=1
WHERE r.role_id IN ('Role_Sys_Admin','Role_District_Admin','Role_School_Admin','Role_Researcher','Role_Teacher','Role_Student','Role_Parent');
```

> 注：上面的 SQL 只是模板，正式执行时建议按 `menu_code` 白名单逐条落库，避免“全量交叉授权”。

## 7. 分期范围与验收目标

## Phase 1：基础权限模型建设

### 目标

把“权限是什么、有哪些资源、怎么查出来”先建立起来。

### 交付物

1. 页面/菜单目录表
2. 角色权限表
3. 权限查询接口
4. 权限目录初始化数据
5. 页面元数据约定

### Phase 1 完成标准

- 系统内所有基础页面都能被登记
- 每个页面都有稳定业务编码 `menu_code`
- `menu_id` 为数字主键
- 可以查到某个角色当前拥有哪些 `PAGE_{menuCode}_READ / WRITE`
- 可以为后续页面授权页提供数据源

---

## Phase 2：可视化授权与生效

### 目标

让超级管理员能够在页面上直接编辑、保存角色权限，并在系统里生效。

### 交付物

1. 角色权限配置页
2. `R/W` 矩阵交互
3. 权限保存接口
4. 菜单/路由/页面权限接入
5. 修改类接口鉴权接入
6. 基础联调与验收脚本

### Phase 2 完成标准

- 超级管理员可编辑角色权限
- 保存后权限立即生效
- 没有权限的菜单不显示
- 没有权限的页面不能进入
- 没有 `W` 的用户不能操作修改接口

## 8. 角色权限台的展示原则

### 8.1 目录展示原则

角色权限页展示的是：

- **系统级全量目录**
- 当前角色的授权状态

而不是：

- 某个角色的导航菜单拼接结果
- 某个角色实际可见的局部菜单

### 8.2 权限判定原则

- **菜单数量不代表权限高低**
- **最终以角色授权为准**
- 当前角色能否使用某功能，只看授权表和权限码

### 8.3 分组建议

系统级全量目录建议按模块分组：

- 系统设置
- 区域治理
- 校园治理
- 教研业务
- 教师业务
- 学生业务
- 家长业务
- 运维中心
- 通用页面

## 11. 后端接口设计

### 11.1 查询菜单目录

`GET /v2/sys-menu`

返回系统所有页面/菜单节点。

### 11.2 查询角色权限

`GET /v2/sys-role/:roleId/permissions`

返回角色当前的 `R/W` 配置，以及展开后的 `PAGE_{menuCode}_READ / WRITE` 权限码视图。

### 11.3 保存角色权限

`PUT /v2/sys-role/:roleId/permissions`

请求体示例：

```json
{
  "items": [
    { "menuId": 1, "canRead": true, "canWrite": true },
    { "menuId": 2, "canRead": true, "canWrite": false }
  ]
}
```

### 11.4 保存策略

采用整角色覆盖：

1. 校验请求体
2. 删除该角色旧权限
3. 批量插入新权限
4. 返回更新数量

## 12. 权限校验与 `resolvePermissionCodes` 方案

### 12.1 权限码共存原则

`resolvePermissionCodes` 维持现有角色权限码不变，同时新增从 `sys_role_menu_perm` 读取的页面权限码：

- 现有权限码：保留
- 页面权限码：动态生成并合并

### 12.2 动态权限码生成规则

根据角色当前的授权记录生成：

- `PAGE_{menuCode}_READ`
- `PAGE_{menuCode}_WRITE`

例如：

- `PAGE_console_system_users_READ`
- `PAGE_console_system_users_WRITE`

### 12.3 查询接口默认校验规则

- 查询类接口默认要求 `PAGE_xxx_READ`
- 写入类接口默认要求 `PAGE_xxx_WRITE`

### 12.4 兼容策略

- 现有 `PERMISSIONS`、`ROLE_PERMISSIONS_MAP` 保持不动
- 新增页面权限码作为补充
- 角色既可以拥有旧权限码，也可以拥有页面权限码

---

## 13. 前端路由守卫方案

### 13.1 实现方式

使用客户端高阶组件 `withPermission` 或 hook：

- 输入当前路由对应的 `menuCode`
- 读取当前角色权限集
- 判断是否拥有 `PAGE_{menuCode}_READ`
- 无权限则跳转 403 或无权限页

### 13.2 路由守卫原则

- 菜单隐藏不是唯一手段
- 直接输入 URL 也必须被拦截
- 页面级守卫必须和后端权限一致

### 13.3 校验对象

优先按页面的 `menuCode` 做守卫，不建议直接只按路径字符串做最终权限锚点。

## 15. 质量要求与验收标准

### 15.1 功能验收

- 超级管理员可按角色查看权限
- 超级管理员可修改角色的 `R/W`
- 保存后权限立即生效
- 系统菜单会随权限变化
- 页面可进入性会随权限变化

### 15.2 安全验收

- 无 `R` 不可进入页面
- 无 `W` 不可修改数据
- 前端隐藏按钮后，后端仍会拒绝越权请求

### 15.3 维护验收

- 菜单资源手工初始化可维护
- 权限与角色字典解耦
- 权限配置可持续扩展

## 16. 与当前审计文档的结合

建议把现有 `docs/ROLE_AUDIT_AND_ROADMAP.md` 中的角色导航矩阵、业务闭环评分、风险提示同步挂到权限台中，形成“权限 + 业务完成度”双视图。

注意：**菜单数量不代表权限高低，最终以角色授权为准。**

区管、校管、教研员的页面范围应按实际业务授权收敛，而不是按导航展示数量去判断权限强弱。

## 17. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 资源目录不完整 | 权限无法覆盖全部页面 | Phase 1 先做页面清单初始化，并允许后续增量补齐 |
| 保存接口与现有角色逻辑冲突 | 影响已有角色体系 | 角色字典保持只读，不改身份链路 |
| 前端权限与后端权限不一致 | 出现越权或误拦截 | 前后端统一使用同一套权限数据源与编码 |
| 页面元数据不规范 | 新页面无法纳入 | 约束 `menu_code`、路由、页面标识的命名规则 |
| 权限配置过细导致复杂度上升 | 第一版难以按期完成 | 第一版只保留 `R/W` 两级 |
| 导航视图与全量目录混用 | 产生“下级菜单比上级更多”的错觉 | 角色权限页只展示系统级全量目录，授权结果另行展示 |

## 18. 推荐命名与编码规范

### 18.1 菜单编码

建议使用稳定、可读、与路由相关的编码，例如：

- `console_system_users`
- `console_system_roles`
- `console_settings_experiments`

不要直接使用中文作为唯一键。

### 18.2 角色编码

继续沿用现有角色编码风格：

- `Role_Sys_Admin`
- `Role_District_Admin`
- `Role_School_Admin`
- `Role_Researcher`
- `Role_Teacher`
- `Role_Student`
- `Role_Parent`

### 18.3 权限含义

- `can_read = 1`：表示可见、可进入、可查询
- `can_write = 1`：表示可编辑、可保存、可执行写操作

## 19. 第一版结论

第一版建议只做 Phase 1 + Phase 2，就足以满足项目的基本要求。

### 能力边界

- 有资源目录
- 有角色授权
- 有保存接口
- 有菜单显隐
- 有路由守卫
- 有写接口拦截

### 不在第一版做的内容

- 自动收集
- 权限缓存（Redis/JWT）
- 按钮全量细粒度权限树
- 字段级权限
- 数据行级权限
- 复杂审计回放

> 结论：先把最小闭环跑通，后续再慢慢优化。
