# 系统角色功能菜单治理方案（完整版）

> 存档位置：`docs/auth/04-role-menu-governance-plan.md`
>
> 审计目的：作为后续菜单收敛、权限初始化、角色授权、功能合并/重命名的统一依据。
>
> 适用范围：系统管理员、区管、校管、教研员、教师、学生、家长 7 大角色。
>
> 设计前提：
>
> - 现有权限配置台以 `sys_menu` + `sys_role_menu_perm` 为核心。
> - 菜单数量不代表权限高低，最终以角色授权数据为准。
> - 第一版不做自动收集，采用手工维护的系统级全量目录。
> - 角色职责分层：3 层治理 + 2 层业务 + 1 层学习使用。

---

## 0. 总体结论

本方案采用“**系统级全量目录 + 角色授权视图**”的治理模型，核心目标是：

1. 把当前项目中分散的页面、菜单、角色、权限关系统一归档。
2. 按 7 大角色重新梳理职责边界，避免治理权限和教学权限混用。
3. 对当前页面进行保留、暂隐藏、删除三类治理。
4. 为权限配置台提供可执行的默认授权初始化 SQL 思路。
5. 为后续审计提供稳定、可追溯的角色-菜单-权限映射依据。

### 关键原则

- **系统管理员权限最大，但主要用于系统配置、运维和权限治理，不作为普通业务账号使用。**
- **区管负责全区治理，校管负责本校治理。**
- **教研员和教师属于业务执行链，教师只管自己的教学范围。**
- **学生和家长属于使用层，只读与协作为主。**
- **菜单数量不代表权限高低，最终以角色授权为准。**

## 1. 角色分层与职责

### 1.1 治理层

| 角色代码 | 角色名称 | 职责边界 | 典型菜单 |
|---|---|---|---|
| `Role_Sys_Admin` | 系统管理员 | 系统配置、运维、权限分配、全局数据修复 | 用户管理、角色管理、组织管理、运维中心、家长绑定审核 |
| `Role_District_Admin` | 区级管理员 | 管辖全区学校、用户、基础组织，查看区级统计 | 区组织管理、学校管理、用户管理、班级管理（全区）、区级审批 |
| `Role_School_Admin` | 校级管理员 | 管理本校班级、教师、学生，分配教学班级 | 本校班级管理、本校教师管理、学生管理、教学班级分配、学校统计 |

### 1.2 业务层

| 角色代码 | 角色名称 | 职责边界 | 典型菜单 |
|---|---|---|---|
| `Role_Researcher` | 教研员 | 管理实验、题库、教材、教研组、教学资源 | 实验管理、题库管理、教材管理、教研组管理、资源审核 |
| `Role_Teacher` | 教师 | 管理自己的教学班级、课堂、作业、实验 | 我的教学班级、我的课堂、任务中心、实验库（教师视图）、教材使用 |

### 1.3 使用层

| 角色代码 | 角色名称 | 职责边界 | 典型菜单 |
|---|---|---|---|
| `Role_Student` | 学生 | 完成实验、任务，查看成长记录 | 实验库（学生视图）、任务中心、成长足迹、个人中心 |
| `Role_Parent` | 家长 | 查看孩子进度，协助家庭实验，监督学习 | 任务中心（家长视图）、家庭实验室、孩子进度、绑定审核 |

### 1.4 分层说明

- **治理层**负责“组织、人员、权限、基础资源”。
- **业务层**负责“教学内容、教研资源、课堂执行”。
- **使用层**负责“学习、任务、成长、家庭协作”。

这套分层与现有权限配置台完全兼容，只需要用 `sys_menu` 和 `sys_role_menu_perm` 进行初始化和维护。

## 2. 治理目标

### 2.1 必须解决的问题

1. 页面职责和权限边界混乱。
2. 教师页面被错误地套上组织管理权限。
3. 区管 / 校管 / 教研员的页面范围不清晰。
4. 系统管理员菜单过多但职责未明确。
5. 前端权限页缺少默认预设，首次配置成本高。
6. 现有页面中有开发中、待收敛、应隐藏的路由和菜单。
7. 教师实验、学生作品、作业流转缺少审核闭环菜单。

### 2.2 目标输出

- 一份可审计的 7 大角色治理方案。
- 一份系统级全量菜单目录策略。
- 一份“保留 / 暂隐藏 / 删除 / 合并重命名”的治理结果。
- 一份可执行的默认授权 SQL 思路。
- 一份后续实施和验收标准。

## 3. 现有程序现状与问题分析

### 3.1 已具备的能力

当前程序已具备：

- 角色与用户绑定能力
- 登录态中角色识别能力
- 页面权限码基础能力
- 权限配置台页面
- `sys_menu` 与 `sys_role_menu_perm` 的基础模型
- 角色权限保存与刷新链路

### 3.2 目前的治理问题

1. **目录来源混乱**
   - 之前权限页一度把不同角色的导航视图拼接成目录，导致角色之间菜单数量错乱。

2. **命名不够业务化**
   - 例如“教室授课管理”对教师来说更像组织管理，不像教学执行入口。

3. **权限与职责没有统一收敛**
   - 教师页面被误用 `org_manage`。
   - 区管 / 校管 / 教研员的边界没有在菜单层体现出来。

4. **默认授权缺少初始化策略**
   - 管理员首次配置需要大量手工勾选，成本高。

5. **教学审核链条未显式归档**
   - 教师实验需教研员审核后发布到广场。
   - 学生作品需校级管理员审核后发布到广场。
   - 作业布置与批阅需按班级到个人闭环。

### 3.3 结论

当前最需要的不是重做权限台，而是：

- 统一菜单归属
- 统一默认授权
- 统一页面命名
- 统一职责边界
- 把审核闭环纳入菜单治理

## 6. 角色分层与审核闭环补充

### 6.1 审核闭环规则

#### 教师实验 → 教研员审核 → 实验广场发布

- 教师在 `teacher-experiment-manager` 创建实验课程。
- 教师提交审核后，状态变更为“待审核”。
- 教研员在 `review/experiments` 中通过或驳回。
- 通过后实验进入 `experiment-square`（实验广场）对学生/家长展示。

#### 学生作品 → 校级管理员审核 → 实验广场发布

- 学生在任务中心或实验作品入口上传实验作品。
- 上传后状态变更为“待审核”。
- 校级管理员在 `review/student-works` 中通过或驳回。
- 通过后作品进入实验广场展示。

#### 作业布置 → 班级到个人 → 教师批阅 → 学生反馈

- 教师在 `teacher-tasks` 按班级布置作业。
- 作业按班级下发到学生个人任务中心。
- 学生提交后，教师批阅并反馈到个人任务中心。

### 6.2 审核状态建议

- 待审核
- 已通过
- 已驳回
- 已发布
- 已撤回（系统或审核人触发）

### 6.3 业务边界说明

- 教研员只审核教师实验课程，不直接编辑教师实验内容；若需修改，走驳回后重新提交。
- 校级管理员只审核学生作品，不直接编辑学生作品内容。
- 教师只可对自己被分配的班级布置作业；班级列表必须由后端按关联关系过滤。
- 学生上传作品必须自动关联所属班级与学生身份。
- 家长只读查看孩子结果，不代替孩子提交核心学习结果。

## 9. 默认授权 SQL 思路

### 9.1 核心原则

- 不使用“全量交叉授权”。
- 采用按 `menu_code` 白名单逐条插入。
- 初始化时只给默认角色的默认菜单。
- 后续由权限配置台进行细化授权。

### 9.2 可执行示例

```sql
-- 教研员：实验审核
INSERT IGNORE INTO sys_role_menu_perm (seq_id, role_id, menu_id, can_read, can_write, status, create_time)
SELECT CONCAT('RM_', 'Role_Researcher', '_', menu_id), 'Role_Researcher', menu_id, 1, 1, 'y', NOW()
FROM sys_menu
WHERE menu_code = 'review/experiments' OR menu_code = 'review_experiments';

-- 校级管理员：作品审核
INSERT IGNORE INTO sys_role_menu_perm (seq_id, role_id, menu_id, can_read, can_write, status, create_time)
SELECT CONCAT('RM_', 'Role_School_Admin', '_', menu_id), 'Role_School_Admin', menu_id, 1, 1, 'y', NOW()
FROM sys_menu
WHERE menu_code = 'review/student-works' OR menu_code = 'review_student_works';
```

> 注：正式执行时建议按实际 `menu_code` 统一命名，避免路径型编码与业务编码混用。

### 9.3 伪代码式执行策略

1. 读取 `sys_menu`
2. 按 `menu_code` 与角色职责映射表进行匹配
3. 为每个角色生成默认 `can_read` / `can_write`
4. 批量写入 `sys_role_menu_perm`
5. 超级管理员可额外进行全量预设

## 10. 实施步骤

### Step 1：确认角色职责边界

- 先按本方案确认 7 大角色边界
- 尤其确认区管 / 校管 / 教研员 / 教师之间的边界

### Step 2：确认菜单归属

- 统一菜单命名
- 统一归属层级
- 统一默认读写范围
- 把审核菜单纳入系统级全量目录

### Step 3：生成默认权限初始化数据

- 依据 `sys_menu` 当前实际数据生成插入脚本
- 避免重复授权与权限冲突

### Step 4：前端权限页采用默认预设

- 角色选择后加载默认预设
- 管理员按需调整
- 保存即生效

### Step 5：逐步收敛旧页面

- 暂隐藏不成熟菜单
- 重命名语义混乱的页面
- 删除完全废弃路由

### Step 6：实施审核闭环页面

- `review/experiments` 仅教研员可读写
- `review/student-works` 仅校管可读写
- `experiment-square` 作为已发布内容统一入口

## 12. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 系统管理员误用于日常业务 | 数据误操作 | 明确系统管理员仅用于系统配置与运维，必要时做独立运维账号隔离 |
| 区管与校管边界不清 | 学校/班级治理混乱 | 明确区管管学校级实体，校管管校内实体 |
| 教研员与教师功能重叠 | 授权混乱 | 教研员偏资源与业务统筹，教师偏个人教学执行 |
| 家长权限过大 | 代做/越权提交 | 家长仅允许协助与查看，禁止代替学生提交核心学习结果 |
| 初始化菜单过宽 | 默认权限失控 | 默认授权按白名单逐条落库，不做交叉全开 |
| 审核菜单越权 | 违规发布 | `/review/experiments` 仅教研员，`/review/student-works` 仅校管；后端接口强制角色校验 |
| 超管参与业务审核 | 业务权责不清 | 超管默认只读可排查，不作为常规审核主体；审核接口可按业务角色额外限制 |

## 13. 结论

本治理方案适合作为当前系统 7 大角色的统一审计依据与后续实施准则。

### 核心结论

1. 角色分层合理：3 层治理 + 2 层业务 + 1 层使用。
2. 菜单与权限必须分离：数量不代表高低，授权才是最终依据。
3. 页面命名要业务化：教师页面改成“我的教学班级/我的课堂”更合理。
4. 区管 / 校管 / 教研员 / 教师的职责边界必须收敛。
5. 默认权限要按白名单初始化，避免全量误授。
6. 审核闭环必须纳入菜单治理与默认权限初始化。
7. 权限配置台可以直接承载这套方案，不需要改核心模型，只需补充初始化映射与前端预设。
