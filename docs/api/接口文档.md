# bs_lab 后端接口文档（V2）

> 自动生成自后端路由源码（`backend/src/http/routes/*.ts`）。本文档覆盖全部对外 HTTP 接口，约 220 个。
> 路径参数统一以 `:id` 形式表示。请求/响应字段命名约定：URL 与请求体顶层 key 为 snake_case（与库列对齐），前端/后端内部类型为 camelCase。

---

## 1. 概述

- **服务形态**：原生 Node.js HTTP 服务（`backend/src/http/server.ts`），默认监听 `:4100`。
- **对外前缀**：所有接口对外以 `/api/v2/*` 暴露；网关/服务端会将 `/api/v2` 前缀剥离为内部 `/v2/*` 再分发。
- **路由分发**：每个领域一个路由文件，导出 `routeXxx(req)` 函数，内部按 `method + path` 精确/前缀/正则匹配。
- **链路追踪**：每个响应携带 `x-trace-id` 响应头（服务端生成 UUID）。
- **媒体地址预签名**：所有 JSON 响应中的 MinIO/S3 存储 key 字段会被自动转换为预签名可访问 URL（`deepPresignResponse`）。

---

## 2. 认证与鉴权

### 2.1 会话机制

- 采用 **Cookie Session**：登录成功后下发 `v2_access_token`（访问令牌）与 `v2_refresh_token`（刷新令牌）两个 HttpOnly Cookie。
- 服务端默认**不信任**客户端直接传入的身份头（`x-user-id`、`x-role`、`x-org-id`、`x-user-name`）；这些头会在进入业务路由前被清除。
- 服务端从 `v2_access_token` 解析出 actor，注入下游身份头：`x-user-id`、`x-org-id`、`x-role`、`x-school-level-id`。
- `access` 过期时，部分接口（如 `GET /api/v2/auth/profile`）可凭 `v2_refresh_token` 自动续期一次。

### 2.2 开发环境例外

- 当 `ALLOW_HEADER_ACTOR=true` 且 `NODE_ENV !== production` 时，允许信任客户端直接传入的身份头，便于本地联调（Postman 等）。

### 2.3 权限模型

- 角色码（roleId）：`Role_Sys_Admin`、`Role_District_Admin`、`Role_School_Admin`、`Role_Researcher`、`Role_Teacher`、`Role_Student`、`Role_Parent`（兼容旧 slug：`system_admin`/`district_admin`/... ）。
- 功能权限码（示例）：`user_manage`、`org_manage`、`role_manage`、`exp_view`/`exp_create`/`exp_edit`/`exp_publish`/`exp_delete`、`question_create`/`question_audit`、`coursebook_manage`、`task_grade`、`scale_manage`、`system_dict_write` 等。
- 页面级权限码（动态配置）：`PAGE_<page>_READ` / `PAGE_<page>_WRITE`（如 `PAGE_review_student_works_WRITE`）。
- `Role_Sys_Admin`（超级管理员）在多数权限校验中直接放行（bypass）。

## 3. 通用约定

### 3.1 统一响应格式

```json
{
  "success": true,
  "data": { },
  "error": null
}
```

失败响应：

```json
{
  "success": false,
  "data": null,
  "error": { "message": "错误描述" }
}
```

> 说明：`v2-ai`、`v2-class` 等少数路由响应体可能额外携带 `message` 字段；`GET /api/v2/version` 仅返回 `{ success, data }`。SSE 流式接口（`/api/v2/ai/chat/stream`）不走 JSON 包裹。

### 3.2 常见 HTTP 状态码

| 状态码 | 含义 |
|---|---|
| 200 | 成功 |
| 400 | 参数校验失败 / 业务校验不通过 |
| 401 | 未登录或会话失效 |
| 403 | 已登录但权限不足 |
| 404 | 资源不存在 / 路由未命中 |
| 409 | 资源冲突（如角色绑定冲突） |
| 422 | 无法识别的业务实体（如角色编码/组织） |
| 500 | 服务内部错误 |

### 3.3 CORS

- 允许方法：`GET, POST, PUT, PATCH, DELETE, OPTIONS`。
- 允许请求头：`content-type, x-role, x-user-id, x-org-id, x-org-name, x-user-name, x-trace-id, x-tenant-id, x-app-id, x-subject-key, x-school-level-id`。
- 允许携带凭证：`access-control-allow-credentials: true`。

## 5. 错误码

## 6. 调用示例

### 6.1 登录

```bash
curl -X POST https://lab.aliberg.cn/api/v2/auth/login \
  -H "Content-Type: application/json" \
  -c cookies.txt \
  -d '{"loginName":"admin","loginPwd":"******"}'
```

### 6.2 携带会话调用受保护接口

```bash
curl https://lab.aliberg.cn/api/v2/auth/profile -b cookies.txt
```

### 6.3 AI 流式对话（SSE）

```bash
curl -N -X POST https://lab.aliberg.cn/api/v2/ai/chat/stream \
  -H "Content-Type: application/json" \
  -b cookies.txt \
  -d '{"message":"帮我设计一个酸碱中和实验"}'
```
