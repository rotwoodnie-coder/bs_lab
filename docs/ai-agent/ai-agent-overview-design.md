# 宝山实验 AI 智能体概要设计文档

- 文档版本：V1.1
- 状态：概要设计（审计优化版）
- 适用范围：`bs_lab` 宝山实验平台 AI 工作台
- 设计目标：基于云端 API 与现有业务数据库 `bs_exp_data`，构建面向学生、教师、教研员三类角色的结构化 AI 智能体体系。

---

## 1. 项目概述

### 1.1 背景

宝山实验平台已具备较完整的实验业务数据模型，包括实验主表 `exp_msg`、实验步骤 `exp_step`、实验材料 `exp_material`、安全性模型 `exp_security` / `exp_material_security`、作业流转 `exp_homework` / `exp_homework_student`、教研资源 `exp_library` 以及社交互动 `social_like` / `social_collection` / `social_evaluate` 等。

在此基础上，引入 AI 的核心目标不是构建独立聊天产品，而是构建一个“AI 增强业务流”的实验工作台，使 AI 能够参与实验内容生成、审核预审、学情分析与区域教研治理，并最终沉淀为可追溯、可校验、可回写的教学资产。

### 1.2 设计定位

本项目采用“云端模型 + 结构化回写 + 人工确认”的架构模式，重点强调：

- AI 负责生成建议与草稿，不直接写入正式业务数据；
- 所有 AI 结果必须经过结构化解析与业务校验；
- 所有关键操作需由人工最终确认；
- AI 能力必须围绕现有数据库 Schema 落地，避免脱离业务实体空转；
- 草稿态与正式态必须隔离，避免污染正式教学资产。

### 1.3 适用对象

- 学生：实验课题构思、方案生成、过程引导、结果表达
- 教师：实验方案预审、作业批阅辅助、学情分析
- 教研员：课标对标、精品资源筛选、区域趋势洞察

### 1.4 审计优化要点

本版本在既有概要设计基础上，吸收专项审计建议，重点补强以下能力：

- **增量对比与冲突控制**：AI 写回前必须标记 AI 改动与用户手动改动，避免覆盖冲突；
- **异步任务闭环通知**：视频分析、长文本总结完成后，通过站内消息主动通知用户返回确认；
- **材料组合风险识别**：除单材料安全标签外，补充材料组合风险矩阵校验；
- **独立分析报告存储**：学情摘要与教研报告不再依赖 `sys_msg`，统一进入专用分析表；
- **草稿态并行缓存**：AI 草稿统一落到临时缓存表或草稿区，人工确认后再写入正式表；
- **任务 Trace 机制**：新增 AI 任务日志，记录请求、模型、Token、状态与耗时，便于审计与追踪。

## 2. 设计目标

### 2.1 总体目标

构建一个面向实验教学全流程的 AI 智能体工作台，覆盖“生成、审核、执行、反馈、治理”五类核心场景，实现以下目标：

1. 提升实验方案生成效率；
2. 提升教师审核与批阅效率；
3. 强化实验安全预审能力；
4. 形成可复用的实验教学资产；
5. 支持教研员进行区域资源治理与课标分析；
6. 建立任务审计、回调通知、草稿确认闭环。

### 2.2 设计原则

1. **Schema 优先**：AI 输出必须映射到现有数据库实体。
2. **分层解耦**：模型推理、业务写库、人工确认分层处理。
3. **结构化输出**：统一输出 JSON Schema，降低文本歧义。
4. **异步优先**：视频、图像等长耗时任务采用异步处理。
5. **可审计可回滚**：所有 AI 操作保留任务记录、差异记录与 trace 信息。
6. **角色隔离**：学生、教师、教研员能力与权限边界明确。
7. **增量写入**：AI 结果默认以差量方式应用，避免覆盖人工编辑内容。

## 3. 总体架构

### 3.1 三段式增强架构

系统采用“AI 增强业务流”而非“对话即功能”的设计方式。

#### 3.1.1 AI Orchestrator 编排层

职责：

- 接收前端输入、页面上下文与多模态素材；
- 识别任务类型与用户角色；
- 采用混合检索策略检索教材、实验库、安全库等背景知识；
- 调用合适的云端模型；
- 产出结构化 JSON 草稿；
- 生成任务 trace 与差异草案；
- 执行输入侧与输出侧合规过滤，拦截敏感提示词与违规内容。

#### 3.1.2 Structured Write-back 回写层

职责：

- 校验 AI 输出的字段合法性与业务约束；
- 将结构化结果拆解为业务实体操作；
- 将草稿写入临时缓存表或草稿区，不直接污染正式业务表；
- 记录 AI 修改项与用户手动输入项的差异；
- 支持字段级、条目级确认与回写，允许用户只采纳部分 AI 结果；
- 为正式发布提供数据准备。

#### 3.1.3 Human Review 人工确认层

职责：

- 对 AI 生成内容进行最终确认；
- 支持“应用到编辑器”“确认发布”“驳回修改”等动作；
- 确保高风险业务始终有人审；
- 在异步任务完成后触发通知，要求用户回到工作台确认。

### 3.2 模型路由策略

| 层次 | 任务类型 | 推荐模型 | 主要数据源 |
| --- | --- | --- | --- |
| 意图层 | 分类、指令解析、角色识别 | 轻量模型 | 页面上下文、用户输入 |
| 逻辑层 | 方案生成、安全推断、课标对标 | 高性能推理模型 | `exp_library`、`data_coursebook`、`exp_msg` |
| 视觉层 | 视频/图片分析、关键帧理解 | 多模态模型 | 多媒体文件 |
| 安全层 | 风险标签匹配、危险等级推断、材料组合风险识别 | 推理模型 + 安全知识库 | `data_material_security`、`material_security`、`exp_material_security` |
| 推荐层 | 跨模块联动、意图预测、主动推荐 | 轻量模型 + 规则引擎 | 当前页面、角色、历史行为 |

### 3.3 RAG 检索策略

- 主知识源优先级：`data_coursebook_unit` > `exp_library` > `exp_msg` > 安全库；
- 实验生成时优先检索同学科、同学段、同年级、同单元内容；
- 安全预审时强制检索与材料标签匹配的安全规则；
- 跨模块联动时结合用户当前页面、角色与最近行为构建动态上下文；
- 默认采用混合检索（关键词 + 向量）以降低幻觉与漏召回风险；
- RAG 结果默认只作为“候选参考”，需经过规则校验后才进入草稿。

### 3.4 结果有效性反馈

任务日志需记录结果是否被采纳，用于后续模型与提示词优化。

### 3.5 系统组件关系

```text
前端工作台
  ├─ 实验编辑器
  ├─ AI Sidebar
  └─ 异步任务状态栏
        │
        ▼
AI Orchestrator
  ├─ 意图识别
  ├─ RAG 检索
  ├─ 模型路由
  └─ 结构化输出
        │
        ▼
Structured Write-back
  ├─ 字段校验
  ├─ 分表映射
  ├─ 差异对比
  └─ 草稿持久化
        │
        ▼
Human Review
  ├─ 应用到编辑器
  ├─ 确认发布
  └─ 驳回修改
```

## 5. 核心任务流

### 5.1 实验方案生成流

1. 用户输入自然语言需求；
2. Orchestrator 识别任务类型与适用角色；
3. 检索 `data_coursebook_unit`、`exp_library` 等背景知识；
4. 模型生成结构化实验草稿；
5. 回写层写入草稿缓存区；
6. AI Sidebar 展示差异与预览；
7. 用户确认后同步到正式编辑内容。

### 5.2 安全预审流

1. 读取实验材料与步骤；
2. 检索安全知识库；
3. 分析潜在风险、单材料风险与材料组合风险；
4. 输出风险标签、危险等级与修改建议；
5. 在教师确认后更新安全相关记录；
6. 在输入侧与输出侧增加敏感词和合规性二次过滤，防止危险化学品制作、违规表达或不当引导进入正式内容。

### 5.3 作业初评流

1. 读取 `exp_homework_student` 与答题记录；
2. AI 生成批阅建议和常见错误总结；
3. 教师确认后形成正式批注；
4. 任务完成后触发站内消息通知；
5. 聚合输出班级学情摘要并存入分析表。

### 5.4 教研分析流

1. 汇总实验库、教材和社交数据；
2. 识别课标覆盖盲区与资源热点；
3. 生成区域治理建议与推荐榜单；
4. 输出教研分析报告；
5. 支持历史纵向对比。

### 5.5 跨模块联动流

AI Orchestrator 可基于当前页面、角色与上下文主动推荐相邻功能，形成跨模块闭环：

- **实验-作业联动**：当实验质量较高或难度较大时，自动推荐关联题目并生成课后强化练习草稿，回写到 `exp_homework`；
- **实验-社交联动**：当学生处于课题具象化阶段时，优先推荐高质量案例与同主题优秀作品，增强启发性与参考价值；
- **耗材-安全联动**：当生成 `exp_material` 清单时，同步校验库存、危险等级与安全规程，必要时触发高亮提示与确认流程；
- **教研-资源联动**：教研员浏览 `exp_library` 或课标页面时，自动推送覆盖度分析、资源缺口与优质案例推荐。

跨模块联动遵循“非侵入式推荐”原则，默认以轻量浮窗、提示气泡或侧边栏卡片呈现，用户点击后再进入详细处理。

## 6. 数据库映射概要

### 6.1 方案生成映射

- 实验标题、原理、说明 → `exp_msg`
- 步骤列表 → `exp_step`
- 材料列表 → `exp_material`
- 结果描述 → `exp_result`
- 参考信息 → `exp_reference`
- 草稿差异与回写痕迹 → `ai_task_draft`
- 典型方案生成 Schema 示例应在 M1 前冻结并纳入接口约束。

### 6.2 安全评估映射

- 实验整体安全等级 → `exp_security`
- 材料实例安全标签 → `exp_material_security`
- 材料组合风险矩阵 → `ai_safety_matrix`
- 安全提示消息 → `sys_msg`

### 6.3 作业批阅映射

- 批阅意见草稿 → `exp_homework_student`
- 学情摘要 → `ai_analysis_report`
- 异常日志与追踪 → `sys_log`
- 任务关联状态 → `ai_task_log.context_ref_id`
- 异步完成通知 → `sys_msg`

### 6.4 教研分析映射

- 课标覆盖度报告 → `ai_analysis_report`
- 热门资源与推荐理由 → 由 `social_*` 与 `exp_library` 聚合生成

## 7. 草稿与任务审计设计

### 7.1 草稿态存储策略

AI 生成结果不直接写正式表，优先写入并行临时表或缓存表，待人工确认后再迁移至正式业务表。

推荐草稿表：

- 草稿数据应采用扁平化条目列表或可展开的 JSON 节点结构，便于前端 diff 渲染与条目级确认；
- 对 `exp_step` 建议以 `order` 对齐，对 `exp_material` 建议以 `name` 对齐进行前后版本比对；
- 冲突时保留用户最新修改，AI 建议进入待确认队列。

推荐草稿表：

- `ai_task_draft`：保存 AI 生成结果、目标实体、草稿 JSON、差异信息与确认状态；
- `ai_task_log`：保存任务执行 trace、模型信息与 token 用量；
- `ai_analysis_report`：保存正式分析型结果，例如学情报告、教研分析报告；
- `ai_safety_matrix`：保存材料组合风险与风险推断结果。

### 7.2 任务 Trace 机制

建议在 `bs_exp_data` 中新增 `ai_task_log`，记录以下字段：

- `request_id`
- `context_ref_id`
- `user_id`
- `task_type`
- `model_used`
- `prompt_tokens`
- `completion_tokens`
- `status`
- `trace_time`
- `error_message`
- `is_accepted`
- `reject_reason`
- `user_feedback_score`

其中 `reject_reason` 建议在详细设计阶段预定义枚举值，例如：`unsafe_content`、`irrelevant`、`format_error`、`other`。

任务日志用于：

- 排查模型调用失败；
- 追踪一次 AI 结果的生成路径；
- 统计成本与 token 消耗；
- 分析不同模型的效果差异；
- 关联具体业务上下文并回写对应实验或报告条目；
- 支持跨模块联动时的上下文回溯与就绪状态回写；
- 形成结果有效性与用户采纳率闭环。

### 7.3 差异对比机制

Structured Write-back 阶段需支持“增量对比”：

- 标记 AI 新增字段；
- 标记 AI 修改字段；
- 标记用户手动修改字段；
- 冲突时优先保留用户最近一次手动输入；
- “最近一次”建议以保存时的 `last_modified` 或版本号为准；
- 在 UI 中高亮冲突位置，避免覆盖。

#### 7.3.1 M1 差异对比选型

M1 阶段采用基于列表索引的简单比对方案：

1. 先按 `order` 对齐步骤列表，按 `name` 对齐材料列表；
2. 再比较每项内容是否新增、删除或修改；
3. 对无法稳定对齐的嵌套对象，采用整体覆盖 + 人工确认。

后续若步骤顺序允许自由调整，可升级为 Myers diff 或 patience diff。

## 8. 前端交互概要

### 8.1 页面结构

采用“编辑器内嵌 AI”模式：

- **主区**：实验编辑器，负责展示与编辑业务表内容；
- **右栏**：AI 建议流、风险提示、操作卡片；
- **底栏**：异步任务进度与状态；
- **工具区**：一键生成、一键预审、一键同步、一键发布。

### 8.2 交互原则

- AI 输出先预览，再应用；
- 支持局部字段回填；
- 不覆盖用户已有修改；
- 支持字段级、条目级勾选确认；
- 支持重新生成同一输入内容；
- 长耗时任务采用异步进度反馈；
- 高风险内容必须弹出确认；
- 异步完成后必须主动通知用户回到工作台确认；
- AI 任务完成后需在关联业务条目上同步就绪状态与提示标记；
- 跨模块联动采用非侵入式浮窗、气泡或侧边卡片呈现，默认不打断当前操作；
- AI 生成内容可通过颜色高亮区分来源，便于用户识别哪些内容来自 AI。

## 9. 安全模型增强设计

### 9.1 单材料风险

沿用现有 `data_material_security`、`material_security`、`exp_material_security` 体系，AI 识别单个材料的危险标签与等级。

### 9.1.1 风险来源约束

M1 阶段优先采用人工预定义规则表作为安全判断依据，规则可先存储在 `ai_safety_matrix`；M3 后再逐步引入模型推理扩展，避免早期误判影响教学安全。

#### 9.1.1.1 `ai_safety_matrix` 字段建议

```sql
CREATE TABLE ai_safety_matrix (
    id BIGINT PRIMARY KEY,
    rule_id VARCHAR(32) NOT NULL,
    material_combination JSON NOT NULL,
    risk_type VARCHAR(64),
    risk_level ENUM('medium','high','critical'),
    description TEXT,
    suggestion TEXT,
    enabled BOOLEAN DEFAULT TRUE,
    created_at DATETIME
);
```

建议说明：

- `rule_id` 用于稳定引用与后续版本演进；
- `material_combination` 用于存储触发规则的材料组合；
- `risk_type` 与 `risk_level` 便于聚合统计与 UI 展示；
- `description` 与 `suggestion` 用于解释与操作建议；
- `enabled` 便于规则热更新与灰度开关；
- `created_at` 用于审计与版本追踪。

### 9.2 材料组合风险矩阵

在单材料风险之外，增加“材料组合风险矩阵”逻辑：

- 识别酸碱混合、氧化还原反应、易燃物与火源组合等高风险模式；
- 风险判断结果写入 `ai_safety_matrix`；
- 高风险组合触发更高等级预警，并建议人工复核。

### 9.3 风险处理原则

- 任何高风险组合必须人工确认；
- AI 仅能给出建议，不得绕过安全审核；
- 安全预审结果需可追溯至具体材料与步骤。

- 任何高风险组合必须人工确认；
- AI 仅能给出建议，不得绕过安全审核；
- 安全预审结果需可追溯至具体材料与步骤。

## 10. 非功能性要求

### 10.1 响应与性能

- AI 对话与建议流响应应保持在可接受范围内；
- 视频分析、图像分析等任务异步执行；
- 任务失败需支持重试与状态恢复；
- 任务完成后应主动通知用户；
- 需要定义用户调用配额、角色级限流与降级策略，避免高频批量调用导致成本失控；
- 限流阈值建议支持动态配置，可放在配置中心或环境变量中，便于上线后调优；
- 可选增强特性：支持 AI 生成内容版本化与草稿链式迭代（例如 `parent_draft_id`），便于回溯与对比，但不影响 M1 核心链路。

#### 10.1.1 初始限流与配额建议

| 角色 | 每日调用上限 | 并发限制 | 排队超时 |
| --- | --- | --- | --- |
| 学生 | 20 次 | 5 请求/分钟 | 30 秒 |
| 教师 | 50 次 | 10 请求/分钟 | 30 秒 |
| 教研员 | 100 次 | 20 请求/分钟 | 60 秒 |

触发限流后应返回友好提示，并引导用户进入异步队列或稍后重试；高峰期优先保障教师审核与课堂教学相关请求。

### 10.2 安全性

- 角色权限严格隔离；
- AI 生成内容必须人工确认后才能发布；
- 不向无关外部网络暴露模型接口；
- 学生数据与实验内容需保留审计链路；
- 学生 AI 仅可写入自身草稿，不可访问他人数据；
- 教师 AI 预警仅限本班范围；
- 教研员 AI 不接触具体学生或教师身份信息；
- 安全提示文案采用“潜在风险提示”，避免绝对化禁止表达。

### 10.3 可扩展性

- 支持追加新的任务类型；
- 支持新增模型与 RAG 数据源；
- 支持未来接入更多多模态能力；
- 支持按角色扩展新的智能体能力。

## 12. 风险与约束

| 风险 | 说明 | 应对策略 |
| --- | --- | --- |
| 模型输出不稳定 | 非结构化输出存在歧义 | 统一 JSON Schema 与字段校验 |
| 安全误判 | 材料或步骤风险识别偏差 | 引入人工确认、组合风险矩阵与二次审核 |
| 数据污染 | AI 结果误写正式数据 | 采用草稿态、临时表与待确认态 |
| 长耗时任务影响体验 | 视频分析等任务耗时较长 | 异步化、状态回调与进度反馈 |
| 角色越权 | 不同身份能力混用 | 基于角色的权限控制 |
| 编辑器覆盖冲突 | AI 回写与人工输入同时发生 | 引入增量对比与冲突高亮 |

## 13. 结论

本概要设计围绕宝山实验平台现有数据库 Schema 构建 AI 智能体体系，采用“三段式增强架构 + 模型路由 + 结构化回写 + 人工确认”的方式，将 AI 能力与实验业务深度融合。结合审计意见后，方案进一步补强了草稿隔离、增量对比、任务 trace、主动通知与材料组合风险控制等机制，能够在不破坏现有业务结构的前提下，逐步实现实验生成、教学审核、学情分析与教研治理的智能化升级。

### 13.1 轻量版边界说明

在 10–15 万预算约束下，本阶段仅建议保留以下能力：

- 学生实验方案生成；
- 基础安全预审；
- 草稿确认与拒绝；
- 任务日志与采纳反馈；
- 最小前端交互界面；
- 教师查看与手工审核入口；
- 教研能力仅保留后续扩展接口。

暂不纳入本阶段的能力包括：

- 组合风险矩阵；
- 视频多模态分析；
- 复杂推荐系统；
- 复杂 diff 编辑器；
- 多模型路由编排；
- 多角色深度自动审核闭环。
