# 数据库开发前规范与字段设计审核稿
## 正式提交版

**版本**：V2.0  
**用途**：架构审核 / 开发前冻结  
**范围**：数据库字段设计、表结构边界、枚举规范、外键关系说明、开发前必须冻结项  
**说明**：本文档依据已确认的 Excel / CSV 原始内容整理，并结合架构审计意见完成最终冻结建议。本文档仅用于开发前审核，不包含 SQL 执行脚本。

---

## 一、文档目的

本文档用于在正式进入开发前，对项目数据库设计进行统一审核与冻结，确保：

1. 数据库结构来源明确
2. 字段语义清晰
3. 命名风格统一
4. 枚举值可控
5. 模块边界明确
6. 业务闭环完整
7. 后续开发不会因数据库结构频繁变更而失控

本项目数据库设计以当前已确认的 Excel / CSV 文件为唯一基础来源，结合系统运行所需的少量必要外键关系进行构建，不引入与现有业务无关的扩展模型。

---

## 二、项目数据库设计目标

本项目数据库的目标不是建立一个无限扩展的平台，而是构建一个能够支撑当前业务顺利推进的稳定底座，满足以下要求：

1. 支撑实验教学核心流程
2. 支撑作业发布、提交、批改、仲裁流程
3. 支撑题库、答题、判题闭环
4. 支撑互动、收藏、评价、积分激励
5. 支撑组织、人员、角色、课题组等基础主数据
6. 支撑素材统一管理
7. 严格控制表结构扩展边界，防止无约束扩表

---

## 三、设计原则

### 3.1 严格按原始文件建模
所有表、字段均来源于已确认的 Excel / CSV 原始内容，不做无依据新增。

### 3.2 仅允许必要修正
允许修正的内容包括：

- 明显拼写错误
- 大小写统一
- 异常不可见字符清理
- 关系缺口补齐
- 明确存在语义歧义时的最小修正

### 3.3 字段设计必须有业务语义
每个字段都应明确：

- 字段用途
- 取值类型
- 是否必填
- 是否参与关联
- 是否允许为空
- 是否作为展示字段或业务控制字段

### 3.4 外键只做必要关联
外键设计只做系统内核心关联，不做过度约束，不引入与项目范围无关的强依赖。

# 五、开发前必须冻结的地基项

以下内容为本项目**开发前必须完成并冻结**的地基项。

## 5.1 时间字段类型统一
### 冻结要求
所有业务时间字段统一改为数据库原生时间类型，优先使用 `DATETIME`，少量系统字段可使用 `TIMESTAMP`。

### 适用字段示例
- `create_time`
- `send_time`
- `read_time`
- `mark_time`
- `confirm_time`
- `submit_date`
- `begin_time`
- `end_time`
- `initiator_time`
- `judge_time`
- `log_time`
- `expire_date`

### 说明
禁止继续使用 `varchar(20)` 存储业务时间，以保证时间范围查询、排序统计和格式一致性。

---

## 5.2 拼写纠错统一
### 冻结要求
数据库字段名与表名中的明显拼写错误必须在开发前统一修正。

### 适用项
- `vedio` → `video`
- `securtiy` → `security`
- `coments` → `comments`
- `refernece` → `reference`
- `requir_date` → `require_date`
- `Subject_id` → `subject_id`
- `expire‌_date` → `expire_date`
- `mark‌_user_id` → `mark_user_id`

### 说明
拼写错误会直接污染 ORM 实体类、接口字段和前端调用，必须在建表前一次性纠正。

---

## 5.3 审计字段补齐
### 冻结要求
核心主表统一补齐审计字段：

- `create_user_id`
- `create_time`
- `update_user_id`
- `update_time`

### 适用范围建议
- `sys_user`
- `sys_org`
- `data_file`
- `material_msg`
- `exp_library`
- `exp_msg`

### 说明
审计字段用于追溯数据来源、修改责任和历史变更轨迹，是教育类系统数据治理的基础要求。

---

## 5.4 作业快照冻结
### 冻结要求
`exp_homework_student` 必须明确作业快照语义，学生提交时应冻结提交版本，避免后续标准库更新影响历史作业展示。

- `exp_id`：学生副本
- `teacher_exp_id`：老师标准版
