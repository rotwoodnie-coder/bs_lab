# Phase 2-4 移动端审计报告

审计范围：`/m` 路由下所有页面、组件与中间件逻辑。

审计依据：
- `docs/mobile/mobile-plan.md`
- `docs/mobile/mobile-development-roadmap.md`

审计结论摘要：当前移动端已具备基础骨架，但在“角色严格对齐”“学段/绑定级联真实性”“老师端与学生端隔离”“中间件白名单与死循环防护”几个关键点上，仍存在明显偏差。整体更接近可演示原型，而非严格按方案收敛的交付版本。

---

## 1. 角色功能完整性

### 结论：⚠️ 部分缺失

#### 对齐情况
- 学生端、家长端、老师端都有对应页面骨架，且 `/m` 首页与 `/m/profile` 已根据角色做了一定分流。
- 家长端提供了“我的孩子”“材料助手”“安全设置”“通知中心”等模块雏形。
- 老师端首页有“教师视频广场”风格和“引用次数”静态展示。

#### 主要问题
1. **老师端个人中心与方案不完全对齐**
   - 方案要求老师个人中心核心模块为：班级管理、作业/任务、实验库、审核中心（含小法庭待仲裁）、数据统计、设置。
   - 当前实现里老师端虽然有“班级管理 / 作业/任务 / 实验库 / 审核中心 / 数据统计”，但缺少明确的“设置”入口，也没有“实验库 / 小法庭待仲裁”之间的清晰业务分层。

2. **学生端功能串线风险较高**
   - 学生端个人中心中有大量静态快捷入口（`/m/experiments`、`/m/tasks`、`/m/works`、`/m/badges`、`/m/settings`），但当前 `/m` 路由树内并未看到这些页面的完整实现和能力闭环。
   - 方案强调一期核心范围是视频广场、作业闭环、绑定、多孩子切换、基础题库与统计；当前学生端更像“占位菜单 + 静态卡片”。

3. **家长端缺少“老师审核结果”与绑定流程的真实闭环展示**
   - 方案要求家长绑定后由老师审核通过并即时可用；当前 `/m/bind/child` 仅为本地 cookie 模拟提交，没有后端审核/等待态/通知态。

#### 相关文件
- `frontend/src/app/m/page.tsx`
- `frontend/src/app/m/profile/page.tsx`
- `frontend/src/app/m/bind/child/page.tsx`
- `frontend/src/app/m/assistant/page.tsx`

#### 修复建议
- 为学生、家长、老师分别建立更明确的角色壳与菜单白名单，避免同一页面通过静态数据“假装分流”。
- 老师端补齐“设置”与“审核中心”中的小法庭待仲裁标签页入口。
- 家长绑定页增加待审核/已审核状态，不要直接写入已绑定 cookie。
- 把仅存在于菜单中的页面逐步补齐，否则应先隐藏入口，避免功能串线。

## 2. 学段差异化

### 结论：⚠️ 部分缺失

#### 对齐情况
- 首页和个人中心已经区分了小学/中学的视觉倾向：
  - 小学：更高饱和、圆角更大、卡片更活泼。
  - 中学：信息密度更高、卡片更紧凑。
- `/m/layout.tsx` 根据 `schoolLevelId` 选择了 `primary` / `middle` 的背景风格。

#### 主要问题
1. **学段判断过于依赖字符串包含，且缺少真实数据源**
   - 方案要求基于 `v_user_school_stage` 等上下文进行分流。
   - 当前仅在前端通过 `schoolLevelId` 字符串做 `includes()` 判断，且无法从页面代码看出它来自真实后端上下文还是临时 mock。

2. **小学 vs 中学的差异主要停留在样式层，没有真正分流组件体系**
   - 方案要求“布局、图标、色彩、信息密度”明显不同。
   - 当前首页/个人中心确实有视觉差异，但仍大量复用同一套组件结构，仅在 className 与文案上微调，分流深度不足。

3. **家长与老师未做学段差异**
   - 方案明确家长和老师界面不分年龄段，但 AI 回复要适配上下文；当前 AI 页只有通用骨架。

#### 相关文件
- `frontend/src/app/m/layout.tsx`
- `frontend/src/app/m/page.tsx`
- `frontend/src/app/m/profile/page.tsx`
- `frontend/src/components/mobile/BottomNav.tsx`

#### 修复建议
- 将学段判断下沉到统一上下文层，避免在页面内重复使用字符串启发式判断。
- 拆出小学/中学专用展示组件，而不是同一组件通过 className 变体硬切。
- 统一校验 `schoolLevelId` 的来源，确保不是前端临时拼装数据。

## 4. 视频播放安全锁

### 结论：⚠️ 部分缺失

#### 对齐情况
- 已实现语音播报能力：`useVideoSecurity()` 会在支持 `speechSynthesis` 的环境中播报安全提示。
- 播报期间存在蒙层，并显示倒计时。
- 安全锁期间“下一步”与步骤卡片会被禁用。

#### 主要问题
1. **“按钮禁用”实现不完整**
   - 当前只对播放页内的步骤按钮和上下步按钮做了禁用。
   - 方案要求“朗读期间按钮禁用、不可跳过、播完后自动启用”，但没有看到更严格的状态机约束，例如禁用点击进度条、键盘快捷操作或视频原生控制引发的跳步。

2. **解锁逻辑存在与倒计时不同步的风险**
   - `useVideoSecurity()` 依赖语音结束事件 `onend` 解锁，但如果浏览器语音异常中断、页面切后台、音频被系统打断，可能出现状态不一致。
   - 当前虽然有 `onerror` 兜底，但仍以“语音播报组件生命周期”而非“业务状态机”作为核心解锁依据。

3. **方案要求的语音播报蒙层文案与按钮禁用场景更严格**
   - 当前是“安全播报中…”，但没有看到“不可跳过”的显式拦截状态与重试逻辑。

#### 相关文件
- `frontend/src/components/mobile/VideoPlayer.tsx`
- `frontend/src/hooks/useVideoSecurity.ts`
- `frontend/src/app/m/video/[id]/page.tsx`

#### 修复建议
- 将安全锁升级为显式状态机，解锁条件应由“播报结束 + 倒计时归零 + 当前步骤确认完成”共同决定。
- 除步骤按钮外，补充对视频跳转、键盘快捷键、外部锚点跳转的拦截。
- 在语音失败时明确降级到“手动确认后解锁”，避免锁死或误解锁。

## 5. 绑定流程

### 结论：❌ 未实现

#### 对齐情况
- 当前页面确实采用了“校区 → 学段 → 年级 → 班级 → 学生姓名”的表单分步结构。
- 支持某些层级缺失时跳过，例如学校无学段可直接进入年级。

#### 主要问题
1. **不是严格的级联选择**
   - 方案要求级联选择应遵循真实层级：校区→学段→年级→班级→学生姓名。
   - 当前实现里存在硬编码数据，且“校区 2 无学段时直接使用 schoolId 取年级”的逻辑与层级模型不严格一致。

2. **层级缺失跳过逻辑不完整**
   - 只处理了“无学段跳过”，没有系统性覆盖“无年级 / 无班级 / 无学生姓名”的逐级跳过。
   - `handleSelect()` 中存在直接 `setStep(prev => prev + 1)`，但没有构建统一的下一有效层级计算器。

3. **绑定成功逻辑违反方案**
   - 方案要求走绑定申请、老师审核、绑定生效的闭环。
   - 当前 `handleSubmit()` 直接写入 `v2_access_token` 与 `bs_has_binding=1`，等于前端伪造已绑定状态。

4. **绑定页直接绕开 API，违反静态数据优先原则**
   - 当前完全是本地硬编码数据和 cookie 模拟，没有真实绑定查询/提交链路。

#### 相关文件
- `frontend/src/app/m/bind/child/page.tsx`

#### 修复建议
- 用统一的数据结构描述每个层级，并实现“自动跳过缺失层级”的通用函数，而不是在步骤分支里手写特例。
- 绑定提交必须走后端接口，前端只展示申请结果，不直接伪造绑定成功。
- 增加待审核状态、审核通过状态、失败状态三个明确 UI。

## 6. 老师端视频广场与个人中心

### 结论：⚠️ 部分缺失

#### 对齐情况
- 老师端首页和个人中心已经明显区别于学生端：
  - 首页为“教师视频广场”。
  - 首页卡片带“引用次数：静态数据”。
  - 个人中心包含班级管理、作业/任务、实验库、审核中心、数据统计等。
- 首页没有看到“每日挑战”入口，当前与方案一致。

#### 主要问题
1. **“引用次数”是静态展示，不是真实业务字段**
   - 方案要求老师端卡片额外显示引用次数，通常应来自真实 API 或至少统一 mock 层。
   - 当前纯前端写死文案，无法证明和视频实体绑定。

2. **老师端与学生端仍复用了同一视频广场数据结构**
   - 只是通过 `HOME_DATA.teacher` 切换文案与布局，底层仍是硬编码视频列表。

3. **个人中心缺少清晰的老师专属操作层**
   - 虽有“待办事项”，但没有老师专属的即时审核入口、批阅入口或小法庭待仲裁标签页。

#### 相关文件
- `frontend/src/app/m/page.tsx`
- `frontend/src/app/m/profile/page.tsx`

#### 修复建议
- 为老师端建立独立的数据源与卡片字段，不要只靠静态文本伪装差异。
- 在个人中心补充老师专属快捷入口：审核中心、待仲裁、小法庭、批阅结果。
- 保持“每日挑战”仅对学生可见，不要在老师端入口出现。

## 7. 代码隔离性

### 结论：⚠️ 部分缺失

#### 对齐情况
- 当前 `/m` 路由下没有明显直接引入 `admin` 目录组件的证据。
- 移动端组件都挂在 `frontend/src/components/mobile/` 下，整体目录隔离尚可。

#### 主要问题
1. **存在双份路径与重复文件痕迹，增加串线风险**
   - 你当前仓库里存在同名或近似同名文件的双路径痕迹，如 `frontend/src/app/m/page.tsx` 与 `frontend\src\app\m\page.tsx` 在变更列表中同时出现，提示可能存在路径管理混乱。
   - 移动端相关组件也出现大小写/重复路径痕迹，如 `BottomNav.tsx`、`MobileShell.tsx`、`VideoPlayer.tsx`、`ChildSwitcher.tsx` 等重复搜索结果，需确认是否为真实重复文件还是索引残留。

2. **大量页面仍以静态数据代替真实 API**
   - 方案要求静态数据优先原则，但不等于长期前端伪造业务状态。
   - 目前首页、个人中心、绑定页、视频页都大量使用硬编码数据，未来很容易与真实后端结构脱节。

3. **未发现 `/m` 页面实际接入统一 API 层**
   - 从当前代码看，视频广场、绑定、个人中心均未使用统一的查询/提交接口层。

#### 相关文件
- `frontend/src/app/m/page.tsx`
- `frontend/src/app/m/profile/page.tsx`
- `frontend/src/app/m/bind/child/page.tsx`
- `frontend/src/app/m/video/[id]/page.tsx`

#### 修复建议
- 继续保持 `/m` 与 `/admin` 目录物理隔离。
- 为移动端建立统一的 API facade 或 mock facade，禁止页面内直接散落硬编码业务状态。
- 清理重复文件/重复路径，确保单一真源。

## 8. 死循环与白名单

### 结论：❌ 未实现

#### 主要问题
1. **中间件对 `/m/login` 和 `/m/bind/child` 的白名单处理不严谨**
   - 虽然代码里尝试放行，但对 `/m/**` 的主逻辑仍然会对未登录或未绑定家长进行重定向。
   - 这类逻辑若叠加 `redirect` 参数、返回页再进入守卫，容易形成循环跳转。

2. **`isPathAllowed()` 存在逻辑缺陷**
   - 该函数对 `PROTECTED_MANAGEMENT_PREFIXES` 的返回始终为 `true`，逻辑上并未真正做保护性过滤。
   - 这说明中间件的路由判断与业务目标并不一致。

3. **绑定页直接写 token 后回跳，未做状态确认**
   - 由于绑定页直接伪造“已绑定”，返回 `/m` 后容易掩盖真实循环问题。

4. **缺少对 `redirect` 参数的规范化与白名单约束**
   - 当前跳转时将原始 `pathname` 写入 `redirect`，没有限制目标路径范围。
   - 若后续登录页/绑定页使用该参数自动回跳，存在重定向链条风险。

#### 相关文件
- `frontend/src/middleware.ts`
- `frontend/src/app/m/bind/child/page.tsx`

#### 修复建议
- 将白名单逻辑和绑定逻辑拆分成明确的、可单元测试的函数。
- 对 `redirect` 参数只允许站内 `/m/**` 安全路径，且必须排除 `/m/login`、`/m/bind/child`。
- 增加专门的循环防护：同一路径重复重定向时直接放行或返回错误页。
- 修正 `isPathAllowed()` 的无效分支，避免伪保护。

## 最终结论

当前 `/m` 体系已完成外壳级搭建，但严格按照 `mobile-plan.md` 与 `mobile-development-roadmap.md` 审计，仍有两类核心短板：

1. **业务闭环不真实**：绑定、老师审核、安全锁、角色数据大多仍停留在静态模拟层。
2. **守卫逻辑不够严谨**：中间件和重定向策略仍有循环风险，缺少真正可验证的白名单与防回跳机制。

建议优先修复 P0 项，再推进角色功能与学段分流的结构性收敛。
