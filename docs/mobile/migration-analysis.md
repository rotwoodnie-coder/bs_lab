## 2. 内容差异摘要

### 0052 / 0053 旧版
- `0052` 创建了 `sys_menu` 与 `sys_role_menu_perm` 两张表，并初始化了第一批菜单。
- `menu_code` 命名偏“前缀 + 业务域”风格，例如：
  - `console_system_users`
  - `console_settings_experiments`
  - `teacher_my_teaching_class`
  - `student_experiments`
  - `parent_tasks`
- `0052` 首批菜单数量为 **15 个**，主要覆盖：系统治理、运维中心及少量基础业务页。
- `0052` 的权限初始化只给 `Role_Sys_Admin` 做了全量可见/可写，属于“超管先落库”的第一版。
- `0053` 在 `0052` 基础上补充了 **7 大角色默认菜单预设**，但仍沿用旧 `menu_code` 体系。
- `0053` 使用 `SELECT ... FROM sys_menu WHERE menu_code IN (...)` 批量落权限，逻辑上已经进入“按角色职责分配”的阶段，但菜单体系仍是旧风格。

### 0054 / 0055 / 0056 新版
- `0054` 将菜单治理升级为“最终版初始化”，并明确写明：
  - 先写 `sys_menu`
  - 再写 `sys_role_menu_perm`
  - `menu_code` 统一采用 `snake_case`
- 新版 `menu_code` 明显更短、更规范，例如：
  - `dashboard`
  - `profile`
  - `user_management`
  - `parent_bindings`
  - `teacher_classroom`
  - `student_tasks`
  - `family_lab`
  - `experiment_square`
- 新版菜单数量为 **24 个**，覆盖范围比旧版更大，且分层更清晰：
  - 治理层
  - 业务层
  - 教师端
  - 学生端
  - 家长端
  - 审核/资源/课题组
- `0055` 将菜单种子拆成独立文件，仅负责 `sys_menu`。
- `0056` 将角色权限种子拆成独立文件，仅负责 `sys_role_menu_perm`。
- `0056` 的权限逻辑改为更直接的“按白名单菜单码授权”，并用 `INSERT IGNORE` 做幂等初始化。

## 3. menu_code 命名规则变化

### 旧版（0052/0053）
- 偏“模块前缀 + 业务场景”拼接式命名。
- 常见前缀：`console_`、`teacher_`、`student_`、`parent_`、`researcher_`。
- 风格不完全统一，且语义更偏工程实现层。

### 新版（0054/0055/0056）
- 统一为更标准的 `snake_case` 语义命名。
- 更像“业务菜单语义名”，而不是“系统实现路径名”。
- 命名更短，便于前端路由、权限台、菜单展示统一管理。

## 4. 菜单数量变化

- 旧版首批初始化菜单：**15 个**。
- 新版初始化菜单：**24 个**。
- 说明新版不只是重命名，而是扩大并重组了菜单体系，覆盖更多角色场景与治理页面。

## 5. 角色权限分配逻辑变化

### 旧版
- 以 `Role_Sys_Admin` 的全量授权为起点。
- 其后在 `0053` 中为其他角色做默认预设。
- 权限映射依赖旧 `menu_code`，并且旧版更偏“功能集合式”授权。
- `can_read` / `can_write` 的逻辑较粗，更多是按角色是否属于某菜单集合来判断。

### 新版
- 更明确地按角色职责边界分配：
  - 系统管理员
  - 区级管理员
  - 校级管理员
  - 教研员
  - 教师
  - 学生
  - 家长
- `can_read` 默认更统一，`can_write` 只对白名单菜单开放。
- 菜单和权限都更偏“治理中心 + 业务中心 + 端侧入口”的分层结构。
- 使用 `INSERT IGNORE` + 独立 seed 文件，明显是为了增强幂等性和可单独执行性。

## 6. 原因推测

综合文件内容，较大概率是以下几类原因叠加：

1. **菜单体系重构**
   - 从旧的模块式 `console_* / teacher_* / student_*` 体系，重构为更统一的 `snake_case` 业务菜单体系。
   - 目标是让菜单、前端路由、权限台、文档描述更一致。

2. **权限治理升级**
   - 旧版更像“先把表和基础菜单跑起来”。
   - 新版更像“可独立验收、可幂等执行、可按角色边界治理”的正式方案。
   - 权限逻辑从“初始化脚本”升级为“治理规则落库”。

3. **模块化拆分**
   - `0054` 一次性总装。
   - `0055` 只管 `sys_menu`。
   - `0056` 只管 `sys_role_menu_perm`。
   - 说明团队在向“菜单种子”和“权限种子”解耦，方便单独回滚、单独验收、单独补数。

## 7. 冲突与风险判断

### 是否存在冲突
- **存在明显的潜在冲突风险**。
- 旧版与新版的 `menu_code` 完全不是同一套命名体系。
- 如果新旧脚本混用，很可能出现：
  - `sys_menu` 中同时存在两套功能相似但 `menu_code` 不同的菜单
  - `sys_role_menu_perm` 指向旧菜单码或新菜单码，但前端/后端只认其中一套
  - 同一业务页面对应多个菜单记录，造成权限台展示和实际路由不一致

### 具体风险点
1. **重复初始化风险**
   - `0052/0053` 与 `0054/0055/0056` 都在初始化菜单和角色权限。
   - 若未做清理，数据库可能同时保留两套数据。

2. **menu_code 混乱风险**
   - 旧版如 `console_system_users`、`teacher_tasks`
   - 新版如 `user_management`、`teacher_tasks`、`dashboard`
   - 部分名字表面相近，实际不一定是同一功能源，易造成误配。

3. **权限失配风险**
   - 角色权限表可能存在旧菜单码对应权限，但新路由查的是新版菜单码，导致“有权限但进不去”或“页面能进但权限台没记录”。

### 风险等级
- **中高风险**
- 尤其是在“先执行旧版、后执行新版”或“新旧 seed 同时被纳入部署脚本”时，需要重点防止双套菜单共存。

## 8. 结论

这组迁移文件最像一次 **菜单体系重构 + 权限治理升级 + 初始化脚本模块化拆分** 的演进过程。

- `0052/0053`：旧版基础落库与角色默认权限预设
- `0054/0055/0056`：新版正式治理方案，强调 `snake_case`、幂等初始化和 seed 拆分

建议上线时只保留一套初始化路径，并在部署前确认 `sys_menu` 中不存在旧版 `console_*` 体系与新版 `snake_case` 体系并存的情况。
